# Git 提交规范建议

## 一、为什么补这份文档

对于求职展示型项目来说，代码本身很重要，但提交历史也会影响项目给人的第一印象。

如果提交信息过于随意，例如“test”“update”“修改一下”，会让项目看起来像临时拼出来的；如果提交信息清楚、分组明确，项目整体会更像认真整理过的作品。

## 二、当前项目建议使用的提交前缀

推荐优先使用以下前缀：

- `feat`：新增功能或模块
- `fix`：修复问题
- `docs`：补充或修改文档
- `refactor`：重构代码但不改变功能
- `test`：补充测试代码
- `chore`：杂项调整，例如依赖整理、目录清理

## 三、适合当前项目的提交示例

如果你想拆分成多次提交，可以参考下面这种方式：

1. `feat: 完成配置层和请求封装`
2. `feat: 完成用户模块接口自动化测试`
3. `feat: 完成仓库模块接口自动化测试`
4. `feat: 完成 Issue 模块接口自动化测试`
5. `feat: 增加 YAML 数据驱动和数据库校验`
6. `docs: 完善 README 与项目说明文档`
7. `chore: 清理空文件并统一编码格式`

## 四、如果只想快速上传 GitHub

如果当前目标是尽快上传，而不是追求很细的提交历史，那么首个提交也完全可以直接使用一条较完整的信息：

```bash
git add .
git commit -m "feat: 完成 Gitea 接口自动化测试项目"
```

## 五、当前建议

对于你这份项目来说，如果现在还没有正式提交，我建议直接使用一条完整首提交流程就够了。等后续继续补截图、CI 或标签场景时，再按模块做增量提交，这样最省时间，也足够像样。
